Release Life Cycle

  • Agile is Everything!
  • 3 Kinds of Releases
    • Planned Releases
    • Patches
    • the OMG! OMG! OMG! HotFix

Release Life Cycle: Planned Releases

  • Minor and Major Semantic Versions
    • Major Releases break previous versions
    • Minor Releases do not break previous versions
  • Should be scheduled well in advance
  • Should contain new or updated features
  • Can also contain Bug Fixes and Content changes

Release Life Cycle: Patches

  • Patch Semantic Versions
    • Fixes break(s) in previous version
  • Should have a life span of about 24 hours
  • Should only contain bug fixes, not new functionality

Release Life Cycle: Hot Fix

  • Should Count as a Patch
  • Should not be used unless absolutely nessaceary
  • Its better Practice in the event that a HotFix would be required to Roll Back to the last know working version

Release Life Cycle: Best Practices

  • Commit only working and completed code
  • Descriptive Commit Messages
    • Good: "Update SQL to sort by date"
    • Bad: "Updates"
  • Work on Project Branches, not on Release Branches
    • Assume that any Release Branch can go live at any time

Release Life Cycle: Containers!

  • Containers (Docker) allow for us to quickly and easily deploy full working environments distributed across multiple isolated environments
  • Important Notes!
    • Containers are Isolated and Contained - You can not Edit them directly
    • We can run multiple of the same container on the same host or across multiple hosts
    • Each Container can run is in its own runtime environment

Current CI Life Cycle

  • Focuses on Branches
  • Work Flow
    • Merge to Master
    • Tests are Run (if found)
    • Files are Deployed / Pulled
    • Containers are Built, Deployed, and Started (if required)
    • Tag is Created based off Master

Current CI Life Cycle: The Good Stuff

  • Its fairly quick
  • Its fairly predictable
  • Its easy to follow

Current CI Life Cycle: The Bad Stuff

  • Its not versatile
  • Its does not scale horizontally
  • Its does not allow an for easy rollback

New CI Life Cycle

  • Focuses on Tags
  • Work Flow
    • Merge to Master
    • Tests are Run (if found)
    • Tag is Created based off Master
    • Tag is Deployed
    • Containers are Built, Deployed, and Started (if required)

New CI Life Cycle: The Good Stuff

  • Deployment based on Tag means that we can easily revert to any previous tag
  • Scales out better because we can use multiple runners with a simple script

New CI Life Cycle: The Bad Stuff

  • Potentially a little slower
